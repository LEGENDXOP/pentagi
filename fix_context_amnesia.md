# Fix: Context Amnesia / State Reset Loop in PentAGI

## Root Cause Analysis

After reading the source code and flow reports (17, 18, 19), the "context amnesia" is **not a single bug** — it's 5 compounding failures that create a feedback loop. I'll explain each one and then propose concrete fixes.

---

### Root Cause #1: ContextManager is Local and Disconnected from the Message Chain

**The core problem.** `ContextManager` is instantiated as a local variable inside `performAgentChain()`:

```go
ctxManager = NewContextManager(defaultMaxContextTokens)
```

It tracks tool results added via `ctxManager.Add(response, funcName)`, classifies them by priority, and prunes them. **But the pruning results are never applied back to the actual `chain []llms.MessageContent` that gets sent to the LLM.**

Look at the pruning code in the main loop:

```go
if ctxManager.GetItemCount() > recentToolWindowSize {
    ctxManager.ReclassifyByAge()
    stats := ctxManager.Stats()
    if stats.OverBudget {
        prunedItems := ctxManager.Prune()
        logger.WithField("context_stats", stats.FormatStats()).
            WithField("pruned_items", len(prunedItems)).
            Debug("context manager pruned items before LLM call")
    }
}
```

**`prunedItems` is computed and then... logged.** The returned slice is never used to actually modify the `chain`. The chain keeps growing unbounded (from the ContextManager's perspective), and the only thing actually controlling chain size is the `summarizer.SummarizeChain()` call at the bottom of the loop.

**Impact:** The ContextManager's intelligent priority-based pruning (protect findings, summarize old tools, drop noise) is completely dead code from a functional standpoint. The chain summarizer (which is a separate system — `csum.Summarizer`) does its own aggressive summarization that has no concept of "findings priority" or "tool result importance". When it summarizes, it can (and does) destroy critical context.

### Root Cause #2: Chain Summarization Destroys Context Without Preserving State

The `summarizer.SummarizeChain()` at the bottom of the main loop is the **actual** mechanism that prevents the chain from exceeding the LLM's context window. When the chain gets too long, it:

1. Splits the chain into sections via `NewChainAST`
2. Summarizes older sections into compressed text
3. Preserves only the last N sections intact

The problem is in `restoreChain()`:

```go
summarizer := csum.NewSummarizer(csum.SummarizerConfig{
    PreserveLast:   true,
    UseQA:          true,
    SummHumanInQA:  true,
    LastSecBytes:   lastSecBytesAfterRestore,  // 16 KB
    MaxBPBytes:     maxBPBytesAfterRestore,     // 8 KB
    MaxQASections:  maxQASectionsAfterRestore,   // 3
    MaxQABytes:     maxQABytesAfterRestore,      // 20 KB
    KeepQASections: keepQASectionsAfterRestore,   // 1
})
```

When a subtask chain is restored (after timeout/restart), only **1 QA section** is kept intact, with a max of **20 KB** total for QA context. For a pentest subtask that may have had 50+ tool calls, this reduces 95% of the execution history to a brief summary. The summary is generated by another LLM call (`summarizeHandler`) which has no concept of what's important for pentest continuity.

**After summarization, the agent literally doesn't know what it already did.** So it re-reads STATE.json, FINDINGS.md, HANDOFF.md, and starts over.

### Root Cause #3: ExecutionState is Written but Never Injected Back Into the Chain

The `ExecutionState` (STATE.json concept) is persisted to the DB:

```go
if subtaskID != nil {
    phase := "executing"
    if wantToStop {
        phase = "finishing"
    }
    execState.Update(metrics, phase)
    if stateJSON, err := execState.ToJSON(); err == nil {
        stateWriter.Write(*subtaskID, stateJSON)
    }
}
```

And on resume, it's restored:

```go
if loaded := ParseExecutionState(dbSubtask.Context); loaded != nil {
    execState = loaded
    toolCallCount = loaded.ToolCallCount
    metrics.ToolCallCount = loaded.ToolCallCount
    // ... etc
}
```

**But the restored state is only used to restore internal Go counters (`toolCallCount`, `metrics.ErrorCount`).** It is **never injected into the LLM's message chain** as context. The LLM doesn't see "you already made 47 tool calls, you're in the exploitation phase, you already ran nmap 3 times" — it only sees whatever survived chain summarization.

The `ResumeContext` field (generated every 10 tool calls as RESUME.md content) is stored in `execState.ResumeContext` but is also **never injected into the chain on resume**.

### Root Cause #4: getExecutionContextBySubtask Returns Raw DB Context (Not Resume Context)

When the chain is rebuilt on resume, `getExecutionContext()` is called:

```go
executionContext, err := fp.getExecutionContext(ctx, taskID, subtaskID)
```

This calls `getExecutionContextBySubtask()`:

```go
func (fp *flowProvider) getExecutionContextBySubtask(ctx context.Context, taskID, subtaskID int64) (string, error) {
    subtask, err := fp.db.GetSubtask(ctx, subtaskID)
    if err == nil && subtask.TaskID == taskID && subtask.Context != "" {
        return subtask.Context, nil
    }
    return fp.getExecutionContextByTask(ctx, taskID)
}
```

The `subtask.Context` column contains the serialized `ExecutionState` JSON — the raw `{"phase":"executing","tool_call_count":47,...}` object. This is returned as the `executionContext` string.

But `executionContext` is used in two places:
1. Passed to `performReflector()` as the `executionContext` argument (used in template rendering)
2. Used for industry detection via `DetectIndustry(executionContext)` — which scans for keywords like "e-commerce"

When `subtask.Context` contains only the raw ExecutionState JSON, `DetectIndustry()` won't find industry keywords. And the reflector gets a JSON blob instead of a meaningful execution context description. **The execution context for the subtask was overwritten by the state writer with raw state JSON, destroying the original execution context that described the task/subtask relationship.**

### Root Cause #5: The In-Container STATE.json is a Phantom

The flow reports mention that the agent reads `/work/STATE.json` in the Kali container repeatedly but it's always in `"phase": "recon"` with `"recon": "pending"`. This is a **different** STATE.json from the DB-persisted `ExecutionState`.

The in-container STATE.json is written by the **LLM agent itself** via terminal commands (`cat > /work/STATE.json << 'EOF'`). The agent is prompted to maintain it, but:

1. The agent's first priority is always reconnaissance — it creates STATE.json with `"phase": "recon"` 
2. After a context reset (chain summarization), the agent loses memory of having updated the state
3. It reads STATE.json, finds `"phase": "recon"`, and starts recon again
4. The `cat > STATE.json << 'EOF'` heredoc pattern is vulnerable to context deadline truncation (the file gets truncated to 0 bytes when the write is killed mid-stream — this is why the backup/restore mechanism exists)

The DB-side `ExecutionState` (Root Cause #3) tracks progress correctly, but the LLM never sees it. The in-container STATE.json that the LLM can see is never reliably updated because the LLM keeps losing context about having done work.

---

## The Feedback Loop

```
Chain gets long → Summarizer compresses it → Agent loses context
→ Agent reads STATE.json → STATE.json says "recon:pending" 
→ Agent starts recon again → Chain gets long again → REPEAT
```

Meanwhile:
- ContextManager tracks priorities but never applies them to the chain
- ExecutionState tracks progress but never tells the LLM about it
- ResumeContext is generated but never injected into the chain

---

## Proposed Fixes

### Fix 1: Inject Execution State into the Chain on Resume (CRITICAL)

When `performAgentChain` resumes with a loaded execution state, inject a human-role message summarizing what was already done. This is the **single highest-impact fix**.

**File:** `performer.go` — inside `performAgentChain()`, after the execution state restore block

```go
// CURRENT CODE (around line ~180-200 of performAgentChain):
if subtaskID != nil {
    if dbSubtask, err := fp.db.GetSubtask(ctx, *subtaskID); err == nil && dbSubtask.Context != "" {
        if loaded := ParseExecutionState(dbSubtask.Context); loaded != nil {
            execState = loaded
            toolCallCount = loaded.ToolCallCount
            metrics.ToolCallCount = loaded.ToolCallCount
            metrics.ErrorCount = loaded.ErrorCount
            for _, cmd := range loaded.AttacksDone {
                metrics.AddCommand(cmd)
            }
            metrics.LastToolName = loaded.CurrentAttack
            logger.WithFields(logrus.Fields{
                "resumed_tool_calls": loaded.ToolCallCount,
                "resumed_phase":      loaded.Phase,
                "resumed_errors":     loaded.ErrorCount,
            }).Info("resumed execution state from DB")
        }
    }
}
```

**PROPOSED CHANGE — add this block immediately after the above:**

```go
// FIX: Inject resume context into the message chain so the LLM knows
// what has already been done. Without this, chain summarization strips
// execution history and the agent restarts from scratch.
if execState != nil && execState.ToolCallCount > 0 {
    resumeMsg := buildResumeInjectionMessage(execState)
    chain = append(chain, llms.MessageContent{
        Role: llms.ChatMessageTypeHuman,
        Parts: []llms.ContentPart{
            llms.TextContent{Text: resumeMsg},
        },
    })
    logger.WithField("tool_call_count", execState.ToolCallCount).
        Info("injected resume context into chain from persisted execution state")
}
```

**New function to add (in `helpers.go` or `execution_state.go`):**

```go
// buildResumeInjectionMessage creates a human-role message that tells the LLM
// exactly what has already been done, preventing it from re-running completed work.
func buildResumeInjectionMessage(state *ExecutionState) string {
    var sb strings.Builder
    sb.WriteString("[EXECUTION RESUME — READ CAREFULLY]\n\n")
    sb.WriteString(fmt.Sprintf("You are RESUMING a subtask that was interrupted. Here is your progress:\n\n"))
    sb.WriteString(fmt.Sprintf("  Phase: %s\n", state.Phase))
    sb.WriteString(fmt.Sprintf("  Tool calls already made: %d\n", state.ToolCallCount))
    sb.WriteString(fmt.Sprintf("  Errors encountered: %d\n", state.ErrorCount))
    sb.WriteString(fmt.Sprintf("  Last update: %s\n", state.LastUpdate))
    
    if len(state.AttacksDone) > 0 {
        sb.WriteString(fmt.Sprintf("  Tools/attacks already executed: %s\n", strings.Join(state.AttacksDone, ", ")))
    }
    if state.CurrentAttack != "" {
        sb.WriteString(fmt.Sprintf("  Last tool used: %s\n", state.CurrentAttack))
    }

    sb.WriteString("\n⚠️ CRITICAL INSTRUCTIONS:\n")
    sb.WriteString("1. DO NOT re-run nmap, nuclei, subfinder, or any reconnaissance tool that was already executed\n")
    sb.WriteString("2. DO NOT re-read STATE.json, FINDINGS.md, or HANDOFF.md — their content was already processed\n")
    sb.WriteString("3. DO NOT re-install tools (jq, curl, nuclei, etc.) — they are already installed\n")
    sb.WriteString("4. CONTINUE from where you left off — proceed to the next UNFINISHED phase\n")
    sb.WriteString("5. If you completed reconnaissance, move to exploitation immediately\n")
    
    if state.ResumeContext != "" {
        sb.WriteString("\n--- Detailed Resume Context ---\n")
        sb.WriteString(state.ResumeContext)
    }

    return sb.String()
}
```

### Fix 2: Apply ContextManager Pruning to the Actual Chain (HIGH)

The ContextManager's priority-based pruning is well-designed but completely disconnected from the chain. Connect it.

**File:** `performer.go` — replace the current pruning block in the main loop

**CURRENT CODE:**
```go
if ctxManager.GetItemCount() > recentToolWindowSize {
    ctxManager.ReclassifyByAge()
    stats := ctxManager.Stats()
    if stats.OverBudget {
        prunedItems := ctxManager.Prune()
        logger.WithField("context_stats", stats.FormatStats()).
            WithField("pruned_items", len(prunedItems)).
            Debug("context manager pruned items before LLM call")
    }
}
```

**PROPOSED REPLACEMENT:**
```go
if ctxManager.GetItemCount() > recentToolWindowSize {
    ctxManager.ReclassifyByAge()
    stats := ctxManager.Stats()
    if stats.OverBudget {
        prunedItems := ctxManager.Prune()
        // Apply pruning: replace tool result content in the chain with
        // summarized/pruned versions from the context manager.
        chain = applyContextPruningToChain(chain, prunedItems, ctxManager)
        logger.WithField("context_stats", stats.FormatStats()).
            WithField("pruned_items", len(prunedItems)).
            Info("context manager pruned and applied to chain")
        if err := fp.updateMsgChain(ctx, chainID, chain, rollLastUpdateTime()); err != nil {
            logger.WithError(err).Error("failed to update msg chain after context pruning")
            return err
        }
    }
}
```

**New function (add to `context_manager.go`):**

```go
// applyContextPruningToChain replaces tool_result message content in the chain
// with the pruned/summarized versions from the ContextManager. Items that were
// dropped by pruning (PriorityNoise) have their content replaced with a brief
// marker. Items that were summarized have their content replaced with the summary.
func applyContextPruningToChain(
    chain []llms.MessageContent,
    prunedItems []ContextItem,
    cm *ContextManager,
) []llms.MessageContent {
    // Build a lookup: tool result content prefix → pruned content.
    // We use the first 100 chars as a matching key since tool results
    // may have been truncated in the ContextItem.
    type replacement struct {
        content  string
        priority ContextPriority
    }
    replacements := make(map[string]replacement)
    for _, item := range prunedItems {
        if item.ToolName == "" {
            continue
        }
        key := item.ToolName
        if len(item.Content) > 100 {
            key += ":" + item.Content[:100]
        } else {
            key += ":" + item.Content
        }
        replacements[key] = replacement{
            content:  item.Content,
            priority: item.Priority,
        }
    }

    // Walk the chain and replace tool_result content.
    // We also build a set of ALL original tool contents (pre-pruning) so we
    // can detect items that were dropped entirely (not in prunedItems).
    for i := range chain {
        if chain[i].Role != llms.ChatMessageTypeTool {
            continue
        }
        for j := range chain[i].Parts {
            resp, ok := chain[i].Parts[j].(llms.ToolCallResponse)
            if !ok {
                continue
            }
            key := resp.Name
            if len(resp.Content) > 100 {
                key += ":" + resp.Content[:100]
            } else {
                key += ":" + resp.Content
            }
            if repl, found := replacements[key]; found {
                // Replace with pruned/summarized content
                resp.Content = repl.content
                chain[i].Parts[j] = resp
            }
        }
    }

    return chain
}
```

**Caveat:** The matching logic above is approximate — tool result content in the chain may not exactly match what was added to the ContextManager (especially if the ContextManager saw a truncated version). A more robust approach would be to track chain message indices in the ContextItem. See alternative approach in Fix 2b below.

### Fix 2b (Alternative): Index-Based Chain Pruning

A cleaner approach: when adding items to the ContextManager, also record the chain index (position in `[]llms.MessageContent`).

**Changes to `context_manager.go`:**

```go
type ContextItem struct {
    // ... existing fields ...
    
    // ChainIndex is the index into the message chain where this tool result lives.
    // Set to -1 if not associated with a specific chain message.
    ChainIndex int
    
    // ChainPartIndex is the index within Parts of the chain message.
    ChainPartIndex int
}
```

**Changes to `performer.go` — when adding to ctxManager:**

```go
// Track tool result in context manager with chain position for later pruning.
chainIdx := len(chain) - 1  // we just appended the tool result
ctxManager.AddWithChainPosition(response, funcName, chainIdx, 0)
```

Then the pruning application becomes trivial: iterate pruned items, look up chain[item.ChainIndex].Parts[item.ChainPartIndex], replace content.

### Fix 3: Separate Execution Context from Execution State in DB (MEDIUM)

The `subtask.Context` column is currently overwritten by the `AsyncStateWriter` with raw `ExecutionState` JSON. This destroys the original execution context (task/subtask descriptions) that was prepared by `prepareExecutionContext()`.

**Option A: Use a separate DB column for execution state.**

```sql
ALTER TABLE subtasks ADD COLUMN execution_state TEXT DEFAULT '';
```

And update `AsyncStateWriter` to write to the new column instead of `Context`:

```go
// In execution_state.go, update the flush function:
if err := w.db.UpdateSubtaskExecutionState(writeCtx, database.UpdateSubtaskExecutionStateParams{
    ID:             id,
    ExecutionState: data,
}); err != nil {
    // ...
}
```

**Option B (less invasive): Store both in the Context column as a composite JSON.**

```go
// In execution_state.go, change the write to preserve original context:
type SubtaskContextWrapper struct {
    OriginalContext string          `json:"original_context"`
    ExecutionState  *ExecutionState `json:"execution_state"`
}
```

And update `getExecutionContextBySubtask` to extract the original context:

```go
func (fp *flowProvider) getExecutionContextBySubtask(ctx context.Context, taskID, subtaskID int64) (string, error) {
    subtask, err := fp.db.GetSubtask(ctx, subtaskID)
    if err == nil && subtask.TaskID == taskID && subtask.Context != "" {
        // Try to parse as wrapper first
        var wrapper SubtaskContextWrapper
        if json.Unmarshal([]byte(subtask.Context), &wrapper) == nil && wrapper.OriginalContext != "" {
            return wrapper.OriginalContext, nil
        }
        // If it looks like raw ExecutionState JSON, fall back to task-level context
        if state := ParseExecutionState(subtask.Context); state != nil {
            return fp.getExecutionContextByTask(ctx, taskID)
        }
        return subtask.Context, nil
    }
    return fp.getExecutionContextByTask(ctx, taskID)
}
```

### Fix 4: Persist Resume Context to the Container File System (MEDIUM)

The `buildResumeContent()` function generates excellent resume context every 10 tool calls, but it's only stored in `execState.ResumeContext` (DB). The LLM agent looks for `/work/RESUME.md` in the container — and never finds it because nobody writes it there.

**File:** `performer.go` — in the state persistence block, after `execState.ResumeContext = resumeContent`:

```go
// Every 10 tool calls, generate and persist resume context
if toolCallCount > 0 && toolCallCount%10 == 0 {
    resumeContent := buildResumeContent(toolHistory, metrics)
    if resumeContent != "" {
        execState.ResumeContext = resumeContent

        // FIX: Also write RESUME.md to the container filesystem so the
        // agent can find it on restart. Without this, the agent checks
        // for RESUME.md, finds nothing, and restarts from scratch.
        writeResumeArgs, _ := json.Marshal(map[string]interface{}{
            "input":   fmt.Sprintf("cat > /work/RESUME.md << 'RESUME_EOF'\n%s\nRESUME_EOF", resumeContent),
            "timeout": 10,
        })
        if _, writeErr := executor.Execute(ctx, 0, "", "terminal", "", writeResumeArgs); writeErr != nil {
            logger.WithError(writeErr).Warn("failed to write RESUME.md to container (non-fatal)")
        } else {
            logger.WithField("tool_call_count", toolCallCount).
                Info("wrote RESUME.md to container filesystem")
        }
    }
}
```

This bridges the gap: the agent looks for RESUME.md, and now it actually exists.

### Fix 6: Make Chain Summarization Finding-Aware (HIGH)

The `csum.Summarizer` that controls actual chain size has no concept of findings priority. When it summarizes, it can compress a critical vulnerability discovery into "the agent ran some tests and found issues."

**This requires changes in `csum/` package** (not in the files I've reviewed), but the integration point is in `performer.go`. The fix: before chain summarization, extract findings from tool results and inject them as a protected preamble.

**File:** `performer.go` — before the `summarizer.SummarizeChain()` call at the bottom of the main loop:

```go
if summarizer != nil {
    // FIX: Before summarization, extract all findings from tool results
    // and build a findings digest. This will be injected as a system-level
    // context item that the summarizer preserves (it always keeps system + last section).
    findingsDigest := extractFindingsFromChain(chain)
    if findingsDigest != "" {
        // Update the system prompt with a findings section so it survives summarization
        if len(chain) > 0 && chain[0].Role == llms.ChatMessageTypeSystem {
            if text, ok := chain[0].Parts[0].(llms.TextContent); ok {
                // Replace or insert the findings digest in the system prompt
                chain[0].Parts[0] = llms.TextContent{
                    Text: upsertFindingsInSystemPrompt(text.Text, findingsDigest),
                }
            }
        }
    }

    chain, err = summarizer.SummarizeChain(ctx, summarizerHandler, chain, fp.tcIDTemplate)
    // ... existing error handling ...
}
```

**New function:**

```go
// extractFindingsFromChain scans all tool_result messages in the chain and
// extracts lines containing findings keywords. Returns a deduplicated digest.
func extractFindingsFromChain(chain []llms.MessageContent) string {
    var allFindings []string
    seen := make(map[string]bool)
    
    for _, msg := range chain {
        if msg.Role != llms.ChatMessageTypeTool {
            continue
        }
        for _, part := range msg.Parts {
            resp, ok := part.(llms.ToolCallResponse)
            if !ok {
                continue
            }
            findings := ExtractFindings(resp.Content)
            if findings == "" {
                continue
            }
            // Deduplicate by first line
            firstLine := strings.SplitN(findings, "\n", 2)[0]
            if !seen[firstLine] {
                seen[firstLine] = true
                allFindings = append(allFindings, findings)
            }
        }
    }
    
    if len(allFindings) == 0 {
        return ""
    }
    
    return strings.Join(allFindings, "\n---\n")
}

// upsertFindingsInSystemPrompt replaces or inserts the <confirmed_findings>
// block in the system prompt. This block survives chain summarization because
// the system prompt is always preserved.
func upsertFindingsInSystemPrompt(prompt string, findings string) string {
    startTag := "<confirmed_findings>"
    endTag := "</confirmed_findings>"
    
    block := fmt.Sprintf("%s\n%s\n%s", startTag, findings, endTag)
    
    startIdx := strings.Index(prompt, startTag)
    endIdx := strings.Index(prompt, endTag)
    if startIdx >= 0 && endIdx > startIdx {
        return prompt[:startIdx] + block + prompt[endIdx+len(endTag):]
    }
    
    return prompt + "\n\n" + block
}
```

## Fix Priority and Implementation Order

| # | Fix | Impact | Effort | Priority |
|---|-----|--------|--------|----------|
| 1 | Inject execution state into chain on resume | **Critical** — directly stops the restart loop | Low (add 1 function + 10 lines) | **P0** |
| 4 | Write RESUME.md to container filesystem | **High** — gives the LLM a file it already looks for | Low (5 lines) | **P0** |
| 5 | Update in-container STATE.json from Go | **High** — stops "recon:pending" stale state | Low (8 lines) | **P0** |
| 6 | Finding-aware chain summarization | **High** — findings survive context compression | Medium (2 functions) | **P1** |
| 3 | Separate execution context from execution state | **Medium** — prevents context column overwrite | Medium (DB migration or wrapper) | **P1** |
| 2 | Apply ContextManager pruning to chain | **Medium** — makes existing smart pruning functional | High (matching logic is tricky) | **P2** |

### Quick Win Combination

Fixes 1 + 4 + 5 together would eliminate the most damaging pattern (agent restarts from scratch) with minimal code changes (~30 lines of new code + 1 new function). They can be implemented and tested independently.

## Caveats

1. **Fix 4 and 5 execute terminal commands from Go code** — these are tool calls that happen outside the normal agent loop. They won't be visible in the LLM's conversation history. This is intentional (we don't want to pollute the chain with infrastructure updates), but it means the agent won't "remember" writing these files — it'll just find them on read.

2. **Fix 1 depends on the chain being rebuilt on resume.** If the chain is restored via `restoreChain()`, the resume message needs to be injected after the chain is built but before the first LLM call. The current code structure supports this — the execution state restore block runs before the main loop.

3. **Fix 6 modifies the system prompt on every loop iteration** (to keep findings current). This means the system prompt grows over time. This is acceptable because findings are high-value content that SHOULD be in the system prompt, and the system prompt is always preserved by chain summarization. But it means the system prompt could get large if there are many findings — consider capping at 4 KB of findings.

4. **Fix 2 is architecturally the "right" fix** (make the ContextManager actually control the chain) but has the highest implementation risk because the matching between ContextManager items and chain messages is imprecise. Fix 2b (index-based) is cleaner but requires more refactoring.

5. **The in-container STATE.json format** (written by Fix 5) uses the Go `ExecutionState` struct format, which is different from what the LLM agent creates. The LLM creates a more verbose JSON with target-specific state. Consider either (a) teaching the LLM to read the Go format, or (b) having Fix 5 write a hybrid format that includes both the Go-tracked progress AND the LLM-friendly fields.

6. **Thread safety:** `executor.Execute()` is called from inside the state persistence block (Fixes 4, 5) which is outside the normal tool call flow. Make sure this doesn't conflict with any executor state. In practice, the executor is designed for concurrent use, so this should be fine.

---

## Summary

The "context amnesia" is caused by a **disconnect between three systems** that should work together but don't:

1. **ContextManager** — knows what's important but never touches the chain
2. **ExecutionState** — knows what's been done but never tells the LLM  
3. **Chain Summarizer** — controls chain size but destroys important context indiscriminately

The proposed fixes bridge these gaps by:
- Making the Go-side state visible to the LLM (Fixes 1, 4, 5)
- Making important content survive summarization (Fix 6)
- Preventing the execution context from being overwritten (Fix 3)
- Making the ContextManager actually functional (Fix 2)

**Estimated time waste reduction:** Fixes 1+4+5 alone should reduce the 50-70% wasted runtime to ~10-15%, because the agent will know what it already did and won't restart from scratch.
